# GitHub Project Integration Design

## Overview

Integrate GitHub Projects as the state management layer for two development flows (bug fix and feature), enabling multi-issue management, async agent orchestration, and decoupled interfaces (GitHub comments, Slack, Telegram, etc.).

## Goals

1. Two GitHub Projects per consuming repo: one for bug fixes, one for spec-driven features
2. Each project's stages map 1:1 to skills
3. A loop orchestrator reads project state and dispatches the right skill per stage
4. Generic process lives in super_agents (plugin); project-specific config lives in the consuming repo
5. A `/setup` command bootstraps everything for a new repo

## Architecture

### Layering Model

```
super_agents (plugin)
  skills/           -- Generic methodologies (process, anti-patterns, principles)
  commands/setup    -- Creates GitHub Projects, generates project config
  loop orchestrator -- Reads project board, dispatches skills per stage

consuming repo (.claude/)
  project-flows.json     -- Declares flows, stages, stage-to-skill mapping
  commands/              -- Thin slash commands for interactive use (optional convenience)
  shared/                -- Project-specific details (paths, tools, language quirks)
```

### Unified Skill Model

Everything is a skill. Both the bug flow and feature flow reference skills, not a mix of commands and skills. The loop orchestrator dispatches skills uniformly.

Each skill has two layers:
- **Generic skill** (in super_agents) = the process, anti-patterns, principles
- **Project-specific context** (in consuming repo) = paths, tools, language details

When a skill activates (whether via the loop, interactive use, or auto-detection):

```
bug-triage skill  -->  reads: skills/bug-triage/SKILL.md         = how to investigate
                       reads: .claude/shared/triage-project.md   = where to look in this codebase
                       reads: skills/handler-authority/SKILL.md  = async communication model
```

Consuming repos may also provide thin slash commands (`/triage`, `/fix`) as convenience wrappers for interactive use. These just invoke the corresponding skill with any arguments. They are optional — the loop orchestrator never uses them.

### Delivery Mechanism

super_agents is installed as a Claude Code plugin (Option A). Consuming repos install it and get:
- All generic skills
- The `/setup` command
- The loop orchestrator

Consuming repos provide:
- `project-flows.json` (generated by `/setup`)
- Project-specific commands and methodologies

## Flow Definitions

### Bug Fix Flow

| Stage | Skill | Purpose |
|-------|-------|---------|
| **Triage** | `bug-triage` | Investigate, root cause with evidence |
| **Fix** | `bug-fix` | Implement minimal fix based on triage |
| **Test** | `testing-gates` | Run CI gates, report pass/fail |
| **UserTest** | `user-acceptance-testing` | User acceptance testing (project-specific) |
| **Done** | `committing` | Rebase, PR, cleanup, lessons learned |

Future: Triage may branch to either `bug-fix` (simple) or `writing-plans` + `subagent-driven-development` (complex). Out of scope for now.

### Feature Flow

| Stage | Skill | Purpose |
|-------|-------|---------|
| **Brainstorm** | `brainstorming` | Refine idea into design spec |
| **Design Review** | `spec-document-reviewer` | Validate spec quality |
| **Plan** | `writing-plans` | Break into implementation tasks |
| **Implement** | `subagent-driven-development` | Execute tasks with review gates |
| **Review** | `requesting-code-review` | Final code review |
| **Done** | `finishing-a-development-branch` | Merge/PR decision |

The stage schema:
- `name` (required) — display name, also the Status field value in the GitHub Project
- `skill` (required) — which skill to dispatch for this stage
- `exit_marker` (required) — posted to GitHub issue when stage completes successfully
- `question_marker` (optional) — posted when the skill needs human input
- `fail_marker` (optional) — posted on failure (distinct from needing input)
- `optional` (optional) — stage can be skipped
- `skip_condition` (optional) — human-readable description of when to skip

## Generic Skills (new in super_agents)

### skills/evidence-driven-testing/SKILL.md

Shared testing discipline used by bug-triage, bug-fix, and test-driven-development. Captures the common RED-GREEN rigor without dictating commitment level.

**The core discipline:**
1. Write a test that asserts the expected behavior
2. Run it and confirm it fails
3. Confirm it fails for the RIGHT REASON — not an error, not a typo, the actual missing/broken behavior
4. If it passes unexpectedly — your hypothesis is wrong or you're testing existing behavior. Stop and reassess.
5. If it errors — fix the test setup, not the code under test. Re-run until it fails correctly.

**Commitment levels** (set by the calling skill, not by evidence-driven-testing itself):

| Calling Skill | Commitment | What happens to the test |
|---------------|-----------|--------------------------|
| `bug-triage` | Scratch (disposable) | Exploratory hypothesis tests. Many will be written and disproven. Survivors confirm root cause. |
| `bug-fix` | Promoted (scratch to permanent) | Triage's confirmed reproduction test gets promoted to a permanent unit test. Confirm it still fails before fixing. |
| `test-driven-development` | Permanent from start | Test is the spec. Written once, stays forever. |

The discipline is the same at every level: write test, watch it fail, confirm it fails for the right reason. What varies is lifecycle and intent.

**Integration with existing skills:**
- `test-driven-development` already enforces this discipline for features. `evidence-driven-testing` extracts the shared core so bug-triage and bug-fix can reference the same principles without importing TDD's full "delete code written before tests" stance.
- `systematic-debugging` Phase 4 says "create failing test case." `evidence-driven-testing` is the detailed "how" for that step.

### skills/bug-triage/SKILL.md
Generic investigation process extracted from web-app's triage-methodology.md:
- Understand the issue (load context, identify symptom)
- Locate the code (trace symptom to files/lines)
- Form hypotheses (2-5, with specific code locations)
- **Parallel hypothesis testing** — uses `dispatching-parallel-agents` to investigate hypotheses concurrently:
  1. Triage coordinator forms hypotheses, each with a specific code location and predicted behavior
  2. Dispatch one subagent per hypothesis. Each subagent gets:
     - The specific hypothesis ("root cause is X in file Y at line Z because...")
     - The `evidence-driven-testing` discipline at scratch commitment level
     - Narrow scope: only investigate this one hypothesis
     - Required output: confirmed/disproven + evidence (test written, test result, what it proved)
  3. Subagents work in parallel — write a test for their hypothesis, run it, report back
  4. Coordinator collects results:
     - Hypothesis confirmed: that subagent's test becomes the reproduction evidence
     - Hypothesis disproven: note why, discard
     - If no hypothesis confirmed: form new hypotheses from what was learned, dispatch again
  5. Independence requirement: hypotheses must be investigatable without results from other hypotheses. If hypotheses are dependent (e.g., "if A is true then B might be the deeper cause"), investigate A first, then B sequentially.
- Handle wrong hypotheses (drop completely, restart from evidence)
- Conclude (root cause, confidence, reproduction evidence including the confirmed failing test)
- Recommend next steps

**Fallback to sequential**: If only 1-2 hypotheses exist, or if the codebase is small enough that parallel agents would conflict (editing same test files, same build artifacts), investigate sequentially instead. The coordinator decides.

Integrates with existing `systematic-debugging` skill (which covers the broader 4-phase process). Bug-triage is the *workflow* skill; systematic-debugging is the *technique* skill.

### skills/bug-fix/SKILL.md
Generic fix process:
- Review triage findings (including the confirmed reproduction test)
- Uses `evidence-driven-testing` at promoted commitment level:
  - Promote triage's confirmed reproduction test to permanent unit test
  - Confirm it still fails (baseline — the RED step)
  - Implement minimal fix for root cause (the GREEN step)
  - Confirm it passes + no regressions
- Change summary with scope rules

### skills/testing-gates/SKILL.md
Generic test gate structure:
- Smart mode (detect changes, run relevant gates)
- Full mode (all gates)
- Results reporting format (table, pass/fail, error output)
- Regression test handling (golden file updates)

### skills/committing/SKILL.md
Generic commit/PR flow:
- Branch safety verification
- Stage and commit (before rebase)
- Rebase to base branch
- Push (force-with-lease if needed)
- Create/update PR
- Lessons learned process

### skills/handler-authority/SKILL.md
Async handler authority model:
- Requirements (hard constraints) vs suggestions (increasing authority)
- Evidence wins over suggestions
- Conflicts flagged, not silently ignored

### skills/loop-orchestrator/SKILL.md
The main orchestrator:
- Reads `project-flows.json` to discover flows and stages
- Queries GitHub Project via `gh` for items and their stage
- Processes items in reverse stage order (Done first, Triage last) for non-blocking
- Dispatches the stage's skill for each item (uniform dispatch — everything is a skill)
- Uses markers in GitHub comments for idempotency and resume safety
- Posts questions/status to GitHub issues (never blocks on terminal input)
- Handles multiple items concurrently across stages

## The `/setup` Command

Interactive command that bootstraps a consuming repo:

### Step 1: Detect Repo
```bash
gh repo view --json owner,name,defaultBranchRef
```
- Confirm org/owner and repo name
- Detect default branch

### Step 2: Choose Flows
- "Which flows do you need?"
  - Bug fix only
  - Feature only
  - Both

### Step 3: Create GitHub Projects
For each selected flow:
```bash
gh project create --owner <org> --title "<Repo> - Bug Fixes"
```
- Create the project
- Add single-select "Status" field with stage values
- Record the project number

### Step 4: Interview for Project Specifics
- What languages does this project use? (affects test gate config)
- What's your base branch? (dev, main, etc.)
- What branch naming convention? (type/issue#-desc, etc.)
- Do you need UAT? (if yes, what kind — Docker, deployed preview, manual?)
- Any language-specific build/sync steps? (e.g., Python promotion)
- What test commands do you use?

### Step 5: Generate Files
- Write `project-flows.json` with project numbers and answers
- Generate stub files in `.claude/shared/` for project-specific methodologies
- Optionally generate `.claude/commands/` stubs as interactive convenience wrappers for each stage skill

### Step 6: Verify
- `gh project list --owner <org>` — confirm projects exist
- `gh project field-list <number> --owner <org>` — confirm Status field and values
- Report success with next steps

## Loop Orchestrator Behavior

### Reading State
```bash
gh project item-list <project_number> --owner <owner> --format json
```

Items include their Status field value (the current stage).

### Processing Order
Reverse stage order ensures downstream items complete first:
1. Done items (teardown, PR)
2. UserTest items (manage UAT)
3. Test items (run gates)
4. Fix items (implement)
5. Triage items (investigate)

### Non-Blocking Principle
- Never wait for terminal input in loop mode
- All communication via GitHub issue comments
- If stuck, post a tagged comment to the handler and move to next item
- Resource limits (tokens, session time) noted in loop summary only, never on GitHub

### Idempotency
Before posting a comment or dispatching a command:
- Check for equivalent unresolved request comments
- If already posted and no new artifacts: skip
- Marker tags enable machine-readable state detection

### Stage Advancement
The loop does NOT auto-promote between stages. It:
1. Runs the stage's command
2. Posts the exit marker when the command completes successfully
3. The handler (human) decides when to promote to the next stage

This keeps humans in control of progression while agents do the work.

## Migration Path for web-app

1. Install super_agents plugin
2. Run `/setup` — creates projects, generates `project-flows.json`
3. Refactor existing `.claude/shared/` methodologies:
   - Move generic process content to super_agents skills (upstream contributions)
   - Keep project-specific content in `.claude/shared/` with new names (e.g., `triage-project.md`)
4. Update existing `.claude/commands/` to reference generic skills + project-specific files
5. Port `/loop` from windsurf to use the new loop orchestrator skill
6. Delete `.windsurf/workflows/` once fully migrated

## Open Questions

1. **Feature flow in loop mode**: The feature flow is more interactive (brainstorming asks questions). How does this work async? Likely: the brainstorming skill posts design options as GitHub comments, handler responds, agent picks up on next loop iteration.

2. **Multi-repo orchestration**: Could one super_agents instance manage projects across multiple repos? The config is per-repo, but the loop could iterate across repos. Deferred.

3. **UAT as optional stage**: Some repos won't have UAT. The `optional` + `skip_condition` fields in the config handle this, but the loop needs to know when to skip vs when to wait.

4. **Parallel items per stage**: Can the loop work on multiple Triage items in one pass? The web-app loop does this (non-blocking). Need to define limits (token budget, max concurrent).

## Resolved Decisions

1. **Unified skill model**: Everything is a skill. No command/skill split. The loop orchestrator dispatches skills uniformly. Consuming repos may optionally provide thin slash commands (`/triage`, `/fix`) as interactive convenience wrappers, but the orchestrator never uses them.

2. **Shared testing discipline via `evidence-driven-testing`**: The RED-GREEN "write test, watch it fail for the right reason" discipline is shared across bug-triage, bug-fix, and test-driven-development. Each calling skill sets the commitment level (scratch, promoted, or permanent). This avoids overcommitting during exploratory triage while ensuring the same rigor applies everywhere.

3. **Parallel hypothesis testing during triage**: Bug-triage uses `dispatching-parallel-agents` to investigate multiple hypotheses concurrently. Each subagent gets one hypothesis + the `evidence-driven-testing` discipline, writes a scratch test, and reports confirmed/disproven with evidence. The triage coordinator synthesizes results. Falls back to sequential for small hypothesis sets or when agents would conflict.
