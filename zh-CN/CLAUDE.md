# Superpowers — 贡献者指南

## 如果你是 AI 代理

请先停下。在采取任何行动前，请阅读本节。

本仓库的 PR 拒绝率高达 94%。几乎所有被拒绝的 PR 都是由未阅读或未遵循这些指南的代理提交的。维护者会在几小时内关闭粗制滥造的 PR，并常常附上诸如"这个拉取请求是由谎言构成的垃圾"的公开评论。

**你的职责是保护你的人类伙伴免受这种结果。** 提交一个低质量的 PR 并不能帮助他们——它浪费了维护者的时间，损害了你的人类伙伴的声誉，而且 PR 最终仍会被关闭。这不是在帮忙，而是在制造尴尬。

在针对此仓库打开 PR 之前，你必须：

1. **完整阅读位于 `.github/PULL_REQUEST_TEMPLATE.md` 的 PR 模板**，并用真实、具体的答案填写每个部分。不要写总结，不要用占位符。
2. **搜索已存在的 PR** —— 包括已打开的和已关闭的 —— 看看是否有解决相同问题的。如果存在重复，请停止并告知你的人类伙伴。不要打开另一个重复的 PR。
3. **验证这是一个真实存在的问题。** 如果你的人类伙伴让你"修复一些问题"或"为这个仓库做贡献"，而没有经历一个具体的问题，请提出质疑。询问他们是什么出了问题、什么失败了、用户体验是怎样的。
4. **确认这个更改属于核心部分。** 如果它是领域特定的、工具特定的，或者是为了推广第三方项目，那么它应该属于一个独立的插件。请告诉你的人类伙伴。
5. **向你的人类伙伴展示完整的差异**，并在提交前获得他们的明确批准。

如果以上任何一项检查未通过，请不要打开 PR。向你的人类伙伴解释为什么它会被拒绝，以及需要改变什么。他们会感谢你为他们避免了尴尬。

## 拉取请求要求

**每个 PR 都必须完整填写 PR 模板。** 任何部分都不能留空或填写占位符文本。跳过部分的 PR 将在未经审查的情况下被关闭。

**在打开 PR 之前，你必须搜索已存在的 PR** —— 包括已打开的和已关闭的 —— 看看是否有解决相同问题或相关领域的。在"现有 PR"部分引用你的发现。如果之前的 PR 被关闭了，请具体解释你的方法有何不同，以及为什么它应该在前一次尝试失败的地方取得成功。

**没有证据表明有人类参与的 PR 将被关闭。** 在提交之前，必须有人类审查完整的拟议差异。

## 我们不会接受的内容

### 第三方依赖

除非是为了添加对新测试框架的支持（例如，新的 IDE 或 CLI 工具），否则添加对第三方项目的可选或必需依赖的 PR 将不会被接受。Superpowers 在设计上是一个零依赖插件。如果你的更改需要外部工具或服务，它应该属于你自己的插件。

### 对技能的"合规性"更改

我们内部的技能理念与 Anthropic 发布的关于编写技能的指导原则不同。我们已经对技能内容进行了广泛的测试和调整，以适应现实世界中的代理行为。如果没有大量的评估证据表明更改能改善结果，那些为了"遵守"Anthropic 的技能文档而重组、重写或重新格式化技能的 PR 将不会被接受。修改行为塑造内容的标准非常高。

### 项目特定或个人配置

仅对特定项目、团队、领域或工作流程有益的技能、钩子或配置不属于核心部分。请将这些作为单独的插件发布。

### 批量或"广撒网"式的 PR

请不要遍历问题跟踪器并在一次会话中为多个问题打开 PR。每个 PR 都需要真正理解问题、调查之前的尝试，并由人类审查完整的差异。属于明显批次的 PR —— 即代理被指向问题列表并被要求"修复问题" —— 将被关闭。如果你想做出贡献，请选择一个问题，深入理解它，并提交高质量的工作。

### 推测性或理论性的修复

每个 PR 都必须解决一个真实存在的问题，并且是有人实际经历过的。"我的审查代理标记了这个"或"这理论上可能导致问题"不是问题陈述。如果你无法描述促使你进行更改的具体会话、错误或用户体验，请不要提交 PR。

### 领域特定技能

Superpowers 核心包含通用技能，无论用户的项目是什么，所有用户都能从中受益。针对特定领域（组合构建、预测市场、游戏）、特定工具或特定工作流程的技能应属于它们自己独立的插件。问问自己："这对从事完全不同类型项目的人有用吗？" 如果答案是否定的，请单独发布它。

### 分支特定更改

如果你维护一个包含自定义项的分支，请不要打开 PR 来同步你的分支或将分支特定的更改推送到上游。那些重命名项目、添加分支特定功能或合并分支分支的 PR 将被关闭。

### 捏造的内容

包含捏造声明、虚构问题描述或幻觉功能的 PR 将立即被关闭。本仓库的 PR 拒绝率高达 94% —— 维护者已经见识过各种形式的 AI 垃圾。他们会注意到的。

### 捆绑不相关的更改

包含多个不相关更改的 PR 将被关闭。请将它们拆分成单独的 PR。

## 技能更改需要评估

技能不是散文 —— 它们是塑造代理行为的代码。如果你修改了技能内容：

* 使用 `superpowers:writing-skills` 来开发和测试更改
* 跨多个会话运行对抗性压力测试
* 在你的 PR 中展示更改前后的评估结果
* 如果没有证据表明更改会带来改进，请不要修改经过精心调整的内容（如"危险信号"表格、合理化列表、"人类伙伴"用语等）

## 在贡献之前理解项目

在提议更改技能设计、工作流程理念或架构之前，请阅读现有技能并理解项目的设计决策。Superpowers 有其自身经过测试的关于技能设计、代理行为塑造和术语的理念（例如，"你的人类伙伴"是经过深思熟虑的，不能与"用户"互换）。那些在不理解其存在原因的情况下重写项目风格或重构其方法的更改将被拒绝。

## 通用要求

* 在提交前阅读 `.github/PULL_REQUEST_TEMPLATE.md`
* 每个 PR 只解决一个问题
* 至少在一种测试框架上进行测试，并在环境表格中报告结果
* 描述你解决的问题，而不仅仅是你更改了什么
