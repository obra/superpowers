# 完成前验证

## 概述

未经验证就声称工作已完成，这不是高效，而是不诚实。

**核心原则：** 始终先有证据，再做声明。

**违反本规则的字面意思就是违反本规则的精神。**

## 铁律

```
没有新鲜的验证证据，不得声称已完成
```

如果你在本条消息中没有运行验证命令，你就不能声称它通过了。

## 门控流程

```
在声称任何状态或表达满意之前：

1. 确定：什么命令可以证明这个声明？
2. 运行：执行完整的命令（全新、完整地执行）
3. 阅读：完整输出，检查退出码，统计失败数
4. 验证：输出是否确认了该声明？
   - 如果否：陈述实际状态并附上证据
   - 如果是：陈述声明并附上证据
5. 只有这时：才做出声明

跳过任何步骤 = 在撒谎，不是在验证
```

## 常见失败

| 声明 | 需要 | 不充分 |
|------|------|--------|
| 测试通过 | 测试命令输出：0 个失败 | 上次运行的结果、"应该能通过" |
| 代码检查无误 | 代码检查工具输出：0 个错误 | 部分检查、推测 |
| 构建成功 | 构建命令：退出码 0 | 代码检查通过、日志看起来没问题 |
| 缺陷已修复 | 测试原始症状：通过 | 代码已修改、假定已修复 |
| 回归测试有效 | 红-绿循环已验证 | 测试只通过了一次 |
| 代理已完成 | 版本控制差异显示变更 | 代理报告"成功" |
| 需求已满足 | 逐条清单核对 | 测试通过 |

## 危险信号 - 停下来

- 使用"应该"、"大概"、"似乎"等词
- 在验证之前就表达满意（"太好了！"、"完美！"、"搞定了！"等）
- 即将在没有验证的情况下提交/推送/创建 PR
- 信任代理的成功报告
- 依赖部分验证
- 想着"就这一次"
- 疲惫了想赶紧结束
- **任何暗示成功但未运行验证的措辞**

## 合理化借口防范

| 借口 | 现实 |
|------|------|
| "现在应该能用了" | 运行验证 |
| "我很有信心" | 信心 ≠ 证据 |
| "就这一次" | 没有例外 |
| "代码检查通过了" | 代码检查 ≠ 编译器 |
| "代理说成功了" | 独立验证 |
| "我累了" | 疲惫 ≠ 借口 |
| "部分检查就够了" | 部分验证什么也证明不了 |
| "换种说法所以规则不适用" | 精神重于字面 |

## 关键模式

**测试：**
```
✅ [运行测试命令] [看到：34/34 通过] "所有测试通过"
❌ "现在应该能通过了" / "看起来是对的"
```

**回归测试（TDD 红-绿循环）：**
```
✅ 编写 → 运行（通过） → 撤销修复 → 运行（必须失败） → 恢复 → 运行（通过）
❌ "我已经写了回归测试"（没有红-绿验证）
```

**构建：**
```
✅ [运行构建] [看到：退出码 0] "构建通过"
❌ "代码检查通过了"（代码检查不检查编译）
```

**需求：**
```
✅ 重新阅读计划 → 创建清单 → 逐条验证 → 报告差距或完成情况
❌ "测试通过了，阶段完成"
```

**代理委派：**
```
✅ 代理报告成功 → 检查版本控制差异 → 验证变更 → 报告实际状态
❌ 信任代理报告
```

## 为什么这很重要

来自 24 次失败的教训：
- 你的人类伙伴说过"我不相信你"——信任被打破
- 未定义的函数被发布——会导致崩溃
- 遗漏的需求被发布——功能不完整
- 时间浪费在虚假的完成上 → 返工 → 重做
- 违反了："诚实是核心价值观。如果你撒谎，你将被替换。"

## 何时应用

**始终在以下操作之前：**
- 任何形式的成功/完成声明
- 任何表达满意的话语
- 任何关于工作状态的正面陈述
- 提交、创建 PR、完成任务
- 转到下一个任务
- 委派给代理

**规则适用于：**
- 确切的短语
- 改写和同义词
- 暗示成功
- 任何暗示完成/正确的沟通

## 底线

**验证没有捷径。**

运行命令。阅读输出。然后再声称结果。

这是不可商量的。
