# 测试反模式

**在以下情况加载此参考文档：** 编写或修改测试、添加模拟对象、或想要向生产代码添加仅用于测试的方法时。

## 概述

测试必须验证真实行为，而非模拟对象的行为。模拟对象是用来隔离的手段，而非被测试的对象。

**核心原则：** 测试代码做了什么，而不是模拟对象做了什么。

**严格遵循 TDD 可以防止这些反模式。**

## 铁律

```
1. 绝不测试模拟对象的行为
2. 绝不向生产类添加仅用于测试的方法
3. 绝不在不理解依赖关系的情况下使用模拟对象
```

## 反模式 1：测试模拟对象的行为

**违规做法：**
```typescript
// ❌ 错误：测试模拟对象是否存在
test('渲染侧边栏', () => {
  render(<Page />);
  expect(screen.getByTestId('sidebar-mock')).toBeInTheDocument();
});
```

**为什么这是错误的：**
- 你在验证模拟对象是否工作，而不是组件是否工作
- 模拟对象存在时测试通过，不存在时测试失败
- 对真实行为没有任何说明

**你的人类搭档的纠正：** "我们是在测试模拟对象的行为吗？"

**正确做法：**
```typescript
// ✅ 正确：测试真实组件或不要使用模拟
test('渲染侧边栏', () => {
  render(<Page />);  // 不要模拟侧边栏
  expect(screen.getByRole('navigation')).toBeInTheDocument();
});

// 或者，如果侧边栏必须被模拟以实现隔离：
// 不要对模拟对象进行断言——测试 Page 在侧边栏存在时的行为
```

### 检查函数

```
在对任何模拟元素进行断言之前：
  问自己："我是在测试真实组件的行为还是仅仅在测试模拟对象的存在？"

  如果是在测试模拟对象的存在：
    停下来 - 删除该断言或取消模拟该组件

  改为测试真实行为
```

## 反模式 2：在生产代码中添加仅用于测试的方法

**违规做法：**
```typescript
// ❌ 错误：destroy() 仅在测试中使用
class Session {
  async destroy() {  // 看起来像生产 API！
    await this._workspaceManager?.destroyWorkspace(this.id);
    // ... 清理
  }
}

// 在测试中
afterEach(() => session.destroy());
```

**为什么这是错误的：**
- 生产类被仅用于测试的代码污染
- 如果在生产中被意外调用将很危险
- 违反了 YAGNI 原则和关注点分离
- 混淆了对象生命周期和实体生命周期

**正确做法：**
```typescript
// ✅ 正确：测试工具处理测试清理
// Session 没有 destroy() 方法——它在生产中是无状态的

// 在 test-utils/ 中
export async function cleanupSession(session: Session) {
  const workspace = session.getWorkspaceInfo();
  if (workspace) {
    await workspaceManager.destroyWorkspace(workspace.id);
  }
}

// 在测试中
afterEach(() => cleanupSession(session));
```

### 检查函数

```
在向生产类添加任何方法之前：
  问自己："这个方法是否仅被测试使用？"

  如果是：
    停下来 - 不要添加它
    将其放在测试工具中

  问自己："这个类是否拥有该资源的生命周期？"

  如果不是：
    停下来 - 这个方法不应该放在这个类中
```

## 反模式 3：不理解依赖就使用模拟

**违规做法：**
```typescript
// ❌ 错误：模拟破坏了测试逻辑
test('检测重复服务器', () => {
  // 模拟阻止了测试依赖的配置写入！
  vi.mock('ToolCatalog', () => ({
    discoverAndCacheTools: vi.fn().mockResolvedValue(undefined)
  }));

  await addServer(config);
  await addServer(config);  // 应该抛出异常——但不会！
});
```

**为什么这是错误的：**
- 被模拟的方法有测试依赖的副作用（写入配置）
- 为了"安全"而过度模拟破坏了实际行为
- 测试因错误的原因通过或莫名其妙地失败

**正确做法：**
```typescript
// ✅ 正确：在正确的层级进行模拟
test('检测重复服务器', () => {
  // 模拟慢的部分，保留测试需要的行为
  vi.mock('MCPServerManager'); // 只模拟慢的服务器启动

  await addServer(config);  // 配置被写入
  await addServer(config);  // 重复被检测到 ✓
});
```

### 检查函数

```
在模拟任何方法之前：
  停下来 - 先不要模拟

  1. 问自己："真实方法有哪些副作用？"
  2. 问自己："这个测试是否依赖于其中某些副作用？"
  3. 问自己："我是否完全理解这个测试需要什么？"

  如果依赖于副作用：
    在更低层级进行模拟（实际的慢速/外部操作）
    或使用保留必要行为的测试替身
    而不是测试依赖的高层方法

  如果不确定测试依赖什么：
    先用真实实现运行测试
    观察实际需要发生什么
    然后在正确的层级添加最少的模拟

  危险信号：
    - "我模拟这个以防万一"
    - "这个可能很慢，最好模拟它"
    - 在不理解依赖链的情况下使用模拟
```

## 反模式 4：不完整的模拟

**违规做法：**
```typescript
// ❌ 错误：部分模拟——只包含你认为需要的字段
const mockResponse = {
  status: 'success',
  data: { userId: '123', name: 'Alice' }
  // 缺少：下游代码使用的 metadata
};

// 之后：当代码访问 response.metadata.requestId 时会出错
```

**为什么这是错误的：**
- **部分模拟隐藏了结构假设** ——你只模拟了你知道的字段
- **下游代码可能依赖于你未包含的字段** ——静默失败
- **测试通过但集成失败** ——模拟不完整，真实 API 是完整的
- **虚假的信心** ——测试对真实行为没有任何证明

**铁律：** 按照现实中的完整数据结构进行模拟，而不仅仅是你当前测试使用的字段。

**正确做法：**
```typescript
// ✅ 正确：镜像真实 API 的完整性
const mockResponse = {
  status: 'success',
  data: { userId: '123', name: 'Alice' },
  metadata: { requestId: 'req-789', timestamp: 1234567890 }
  // 真实 API 返回的所有字段
};
```

### 检查函数

```
在创建模拟响应之前：
  检查："真实 API 响应包含哪些字段？"

  操作：
    1. 从文档/示例中检查实际 API 响应
    2. 包含系统下游可能消费的所有字段
    3. 验证模拟完全匹配真实响应的结构

  关键：
    如果你在创建模拟，你必须理解整个结构
    当代码依赖于被省略的字段时，部分模拟会静默失败

  如果不确定：包含所有文档记录的字段
```

## 反模式 5：集成测试沦为事后补充

**违规做法：**
```
✅ 实现完成
❌ 没有编写测试
"准备好进行测试了"
```

**为什么这是错误的：**
- 测试是实现的一部分，不是可选的后续工作
- TDD 本可以避免这个问题
- 没有测试就不能声称完成

**正确做法：**
```
TDD 循环：
1. 编写失败的测试
2. 实现使其通过
3. 重构
4. 然后才能声称完成
```

## 当模拟变得过于复杂时

**警告信号：**
- 模拟的准备工作比测试逻辑还长
- 为了让测试通过而模拟所有东西
- 模拟缺少真实组件拥有的方法
- 模拟变更时测试就会中断

**你的人类搭档的问题：** "我们这里真的需要使用模拟吗？"

**考虑：** 使用真实组件的集成测试通常比复杂的模拟更简单

## TDD 防止这些反模式

**为什么 TDD 有帮助：**
1. **先写测试** → 迫使你思考你实际在测试什么
2. **观察它失败** → 确认测试测试的是真实行为，而非模拟对象
3. **最少实现** → 不会有仅用于测试的方法悄悄混入
4. **真实依赖** → 你在模拟之前就能看到测试实际需要什么

**如果你在测试模拟对象的行为，说明你违反了 TDD** ——你在没有先对真实代码观察测试失败的情况下就添加了模拟。

## 快速参考

| 反模式 | 修复方法 |
|--------|---------|
| 对模拟元素进行断言 | 测试真实组件或取消模拟 |
| 生产代码中的仅测试方法 | 移到测试工具中 |
| 不理解就使用模拟 | 先理解依赖关系，最少化模拟 |
| 不完整的模拟 | 完整镜像真实 API |
| 测试沦为事后补充 | TDD——先写测试 |
| 过于复杂的模拟 | 考虑使用集成测试 |

## 危险信号

- 断言检查 `*-mock` 测试 ID
- 方法仅在测试文件中被调用
- 模拟的准备工作占测试的 50% 以上
- 移除模拟后测试就失败
- 无法解释为什么需要模拟
- "以防万一"地使用模拟

## 底线

**模拟对象是用来隔离的工具，而非被测试的对象。**

如果 TDD 揭示了你在测试模拟对象的行为，说明你走偏了。

修复方法：测试真实行为，或质疑你为什么要使用模拟。
