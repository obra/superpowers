## Architecture Overview

The modal system utilizes React components and context for managing the display and state of modals. Configuration data for which modals exist, their properties (like `chatbot_id`), and their associated component paths is stored centrally in the `modal_configurations` Supabase database table.

**Key Components & Data Flow:**

1. **Database (`modal_configurations` table):**
    - The single source of truth for all modal configurations.
    - Contains columns like `modal_key`, `display_name`, `component_path`, `target_page_prefix`, `target_state`, `chatbot_id`, `integration_type`, `interaction_style`, `is_legacy`.
    - Managed via Supabase UI or the Modal Management page (0170).

2. **Service Layer (`client/src/services/modalConfigService.js`):**
    - Provides functions (`fetchModalConfigurations`, `addModalConfiguration`, `deleteModalConfiguration`, `updateModalConfiguration`) to interact with the Supabase table.
    - Handles Supabase client initialization and uses the **user's authenticated session** for data fetching and modification requests (respecting RLS).

3. **Build Script (`scripts/generateModalRegistry.js`):**
    - Runs during the `npm run build` process (as a `prebuild` step).
    - Reads Supabase URL and **Service Key** from the project's root `.env` file.
    - Uses the Service Key to bypass RLS and fetch *all* modal configurations from the database.
    - Generates `client/src/generated/modalRegistry.js`, which exports an object mapping each `modal_key` to a dynamic `import()` statement pointing to the `component_path` from the database.
    - This generated registry is then bundled by Webpack.

4. **Modal Provider (`client/src/components/modal/context/0170-ModalProvider.js`):**
    - Imports and uses the *generated* `modalRegistry.js`.
    - Provides the `openModal` and `closeModal` functions via context.
    - When `openModal(modalKey, modalProps)` is called:
        - It looks up `modalKey` in the generated registry.
        - It dynamically imports the associated component using `React.lazy()` and the import function from the registry.
        - It renders the dynamically imported component within a `React.Suspense` boundary.
        - It passes any `modalProps` provided during the `openModal` call down to the rendered modal component instance.

5. **Modal Triggering (`useModal` hook):**
    - Components import and use the `useModal` hook (`client/src/components/modal/hooks/0170-useModal.js`) to get access to `openModal`.
    - They call `openModal('your_modal_key', { optionalProp1: value1, chatbot_id: 'xyz' })`.
    - The `modal_key` must match an entry in the `modal_configurations` table (and thus in the generated registry).
    - Optional props passed here are received by the modal component instance as `modalProps`.

6. **Modal Management UI (`client/src/pages/0170-modal-management/components/0170-ModalManagementPage.js`):**
    - A React component for viewing and managing configurations.
    - Uses `modalConfigService.js` to fetch configurations directly from the database (using the user's authenticated session) for display.
    - Uses the service to handle adding (`addModalConfiguration`) and deleting (`deleteModalConfiguration`) configurations, respecting RLS policies based on the user's role.
    - **Requires Rebuild:** Adding, deleting, or modifying configurations (especially `modal_key` or `component_path`) requires an application rebuild (`npm run build`) to update the generated `modalRegistry.js`.

The legacy mixin-based system and static registry files (`0200-all-modal-registry.js`) are deprecated.

## Adding a New Modal (Step-by-Step)

Here's the process for adding a new database-configured modal to the application:

1. **Create the Component File:**
    - Create a new React component file for your modal within the appropriate page's modal directory. Follow the naming convention if applicable (e.g., `client/src/pages/2025-quantity-surveying/components/modals/2025-InvoiceHandlingModal.js`).
    - Implement the modal's UI and logic using the base `<Modal>` component and potentially the `useModal` hook for `closeModal`.
    - Ensure the component accepts `modalProps` as a prop to receive data passed during `openModal`.
    - Export the component as default: `export default YourModalComponent;`

2. **Add Trigger Button/Element:**
    - In the page component where the modal should be triggered (e.g., `2025-quantity-surveying-page.js`), add a button or other UI element.

3. **Implement Trigger Logic:**
    - Import the `useModal` hook in the page component: `import { useModal } from '@components/modal/hooks/0170-useModal';`
    - Get the `openModal` function: `const { openModal } = useModal();`
    - In the event handler for your trigger element (e.g., `handleModalClick`), call `openModal` with a **unique string key** for your new modal and any necessary props:

        ```javascript
        const handleModalClick = (modalTarget) => {
          if (modalTarget === 'yourNewModalKey') {
            openModal('yourNewModalKey', { someProp: 'someValue', triggerPage: "PageIdentifier" });
          } // Add other else if branches as needed
        };
        ```

    - **Important:** Do *not* import the modal component directly into the page component for triggering. Rely on the string key and the `ModalProvider`'s registry.

4. **Configure in Database:**
    - Edit the primary SQL setup script for modal configurations (e.g., `sql/setup_modal_configurations.sql`).
    - Add a new `INSERT INTO public.modal_configurations (...) VALUES (...)` statement.
    - Provide values for:
        - `modal_key`: The **exact same unique string key** used in the `openModal` call (e.g., `'yourNewModalKey'`).
        - `display_name`: A user-friendly name for the modal (e.g., `'Handle Invoice'`).
        - `component_path`: The webpack alias path to your component file (e.g., `'@pages/2025-quantity-surveying/components/modals/2025-InvoiceHandlingModal.js'`).
        - `target_page_prefix`: The 4-digit prefix of the page where the modal is primarily used (e.g., `'2025'`).
        - `target_state`: The state group the trigger button belongs to, if applicable (e.g., `'Workspace'`, `'Upsert'`, `'Agent'`).
        - Other fields like `integration_type`, `interaction_style`, `chatbot_id` as needed (default to `'None'` if not applicable).
    - Use `ON CONFLICT (modal_key) DO NOTHING;` to prevent errors if the script is run multiple times.

5. **Update Database:**
    - Execute the updated SQL script (e.g., `sql/setup_modal_configurations.sql`) against your Supabase database instance. This inserts the new configuration row.

6. **Rebuild Application:**
    - Navigate to the `client` directory in your terminal.
    - Run the build command: `npm run build` (or `pnpm build`).
    - This triggers the `prebuild` script (`scripts/generateModalRegistry.js`), which fetches the *new* configuration from the database (using the Service Key) and updates `client/src/generated/modalRegistry.js`.

7. **Test:**
    - Ensure your development server is running (`npm run dev` in `client`).
    - Hard refresh your browser (Cmd+Shift+R or Ctrl+Shift+R) and clear cache if necessary.
    - Navigate to the page and click the trigger button. The modal should now open correctly.

## Build Process: Generating the Registry

A critical part of the system is the build-time generation of the modal registry.

- **Script:** `scripts/generateModalRegistry.js`
- **Trigger:** Runs automatically as a `prebuild` step when executing `npm run build` (defined in `client/package.json`).
- **Credentials:** Reads Supabase URL and **Service Key** from the project's root `.env` file. The Service Key is required to bypass Row Level Security (RLS) and fetch *all* configurations, regardless of who runs the build. Ensure this key is kept secure and is present in the build environment.
- **Function:**
    1. Connects to Supabase using the service key.
    2. Fetches all rows from the `modal_configurations` table where `is_legacy` is false.
    3. Creates a JavaScript object mapping each `modal_key` to a dynamic `import()` statement pointing to the `component_path` specified in the database.
    4. Writes this object to `client/src/generated/modalRegistry.js`.
- **Output (`client/src/generated/modalRegistry.js` - Example):**

  ```javascript
  // THIS FILE IS AUTO-GENERATED BY scripts/generateModalRegistry.js
  // DO NOT EDIT MANUALLY!

  export const modalRegistry = {
    'ContractSetupModal': () => import(/* webpackChunkName: "ContractSetupModal" */ '@pages/0435-contracts-post-award/components/modals/0435-ContractSetupModal.js'),
    'RiskAssessmentModal': () => import(/* webpackChunkName: "RiskAssessmentModal" */ '@pages/2400-safety/components/modals/2400-RiskAssessmentModal.js'),
    // ... other non-legacy modals
  };
  ```

- **Usage:** This generated file is imported by `0170-ModalProvider.js` and used to load the correct modal component when `openModal(modalKey)` is called.

## Modal Management UI (`0170-ModalManagementPage.js`)

This page provides an interface for viewing and managing modal configurations stored in the database.

**Access:**

- Located within the Information Technology (2050) section.

**Key Features & Data Flow:**

- **List Modals:**
    - On load (after auth check), the component calls `fetchModalConfigurations()` from `modalConfigService.js`.
    - This service function uses the **user's current authenticated Supabase session** to fetch data.
    - Supabase applies RLS policies (e.g., `Allow authenticated read access`) based on the user's session.
    - The fetched configurations are displayed in a table.
- **Add New Modal:**
    - Opens the `0170-AddNewModal.js` component.
    - The form collects configuration details (display name, key, component path, target page/state, chatbot ID, etc.).
    - On submit, it calls `addModalConfiguration(newConfigData)` from `modalConfigService.js`.
    - The service sends the data to Supabase using the user's session. RLS policies for INSERT must allow this (e.g., based on user role).
    - **Requires Rebuild:** After adding a configuration, the application must be **rebuilt** (`npm run build`) for the `generateModalRegistry.js` script to include the new modal in the generated registry, making it available to `openModal`.
- **Remove Modal:**
    - Calls `deleteModalConfiguration(modalKey)` from `modalConfigService.js` using the user's session.
    - RLS policies for DELETE must allow this.
    - **Important:** This only removes the configuration from the database. The associated React component file is *not* deleted. The application must be **rebuilt** to remove the modal from the generated registry.
- **Edit Configuration:** (Not Yet Implemented)
    - Would involve an "Edit" form/modal.
    - Would call `updateModalConfiguration(modalKey, updatedData)` from `modalConfigService.js` using the user's session.
    - RLS policies for UPDATE must allow this.
    - Requires a **rebuild** if the `modal_key` or `component_path` changes.
- **Duplicate Modal:**
    - Opens the `0170-DuplicateModal.js` component.
    - Allows selecting an existing modal and defining properties (target page, type, description, display name) for a new copy.
    - Automatically calculates the next sequence number and generates the new `modal_key` and `component_path`.
    - On submit, calls `addModalConfiguration(newConfigData)` from `modalConfigService.js` using the user's session. RLS policies for INSERT must allow this.
    - **Generates Instructions:** After saving the configuration, it displays a prompt with instructions for the next manual steps:
        1. Copying the original component file to the new `component_path`.
        2. Adding the trigger button code snippet (provided in the prompt) to the target page component.
    - **Requires Rebuild:** The application must be **rebuilt** (`npm run build`) after duplication for the `generateModalRegistry.js` script to include the new modal and make it available.
- **Import/Export:** (Not Yet Implemented)
- **View Details:**
    - Opens the `0170-ViewModalDetails.js` component in a modal dialog.
    - Uses the `useModal` hook to call `openModal('ViewModalDetails', { modalData: selectedConfig })`.
    - Displays the details of the selected modal configuration in a read-only format.
    - Does not require a rebuild as it uses a statically registered management modal.

**Implementation Notes:**

- The UI interacts *directly* with the database via the service layer for management tasks (list, add, delete).
- The actual *loading* of modals at runtime relies on the registry generated during the **build process**.
- **Authentication & Authorization:**
    - The build script uses the **Service Key** (highest privilege) to generate the complete registry.
    - The UI uses the **user's authenticated session** for fetching the list and performing add/delete/edit actions.
    - **RLS Policies** in Supabase are crucial for controlling read/write access to the `modal_configurations` table based on the user's role (e.g., `authenticated`, `admin_role_name`). Define policies for SELECT, INSERT, UPDATE, DELETE as needed.
    - Client-side checks in the UI should also prevent unauthorized users from seeing/using management buttons (Add, Remove, Edit).
- **Production Security:** Add/Remove/Edit capabilities should be strictly controlled via RLS and potentially hidden in the UI based on user roles. Ensure the Service Key is kept secure and not exposed client-side.

## Modal Component Implementation

### Receiving Configuration (`modalProps`)

Modal components loaded via this system receive any props passed during the `openModal` call via a single prop named `modalProps`.

```javascript
// Example: client/src/pages/2400-safety/components/modals/2400-RiskAssessmentModal.js
import React from 'react';
import Modal from '@components/modal/components/Modal'; // Base modal wrapper
import { useModal } from '@components/modal/hooks/useModal';

// Destructure modalProps passed from ModalProvider
const RiskAssessmentModal = ({ modalProps }) => {
  const { closeModal } = useModal();

  // Access specific props passed during openModal
  const initialData = modalProps?.initialData;
  const chatbotId = modalProps?.chatbot_id; // Access chatbot_id if passed
  const someOtherProp = modalProps?.someOtherProp;

  console.log("Received modal props:", modalProps);
  console.log("Chatbot ID:", chatbotId);

  const handleSubmit = () => {
    // Use chatbotId or other props in logic
    if (chatbotId) {
      console.log(`Interacting with chatbot: ${chatbotId}`);
      // ... call API or service ...
    }
    // ... other submit logic ...
  };

  return (
    <Modal> {/* Use your base Modal component */}
      <h2>Risk Assessment</h2>
      <p>Chatbot ID (from config/props): {chatbotId || 'N/A'}</p>
      {/* ... form elements using initialData ... */}
      <button onClick={handleSubmit}>Submit</button>
      <button type="button" onClick={closeModal}>Cancel</button>
    </Modal>
  );
};

export default RiskAssessmentModal;
```


## Naming Conventions & Configuration

Modals are configured via the `modal_configurations` database table. To ensure consistency and organization, the following naming convention **must** be followed for both the `modal_key` in the database and the corresponding React component filename:

**Format:** `A-[prefix]-[type]-[sequence]-[description]` (Note: `description` part is used for filename generation, not directly in the key)

- **`A-`:** Static prefix for all modern modal keys.
- **`[prefix]`:** 4-digit number corresponding to the primary page or functional area the modal belongs to (e.g., `0300` for Construction, `0435` for Contracts Post-Award). This should generally match the `target_page_prefix`.
- **`[type]`:** 2-digit code indicating the modal's primary purpose:
    - `01`: Workspace (General UI, data display/entry related to the main page content)
    - `02`: Upsert (Modals specifically for creating or updating data records)
    - `03`: Agent (Modals interacting with external agents like Flowise or n8n)
    - *(Add other type codes as needed)*
- **`[sequence]`:** 3-digit sequential number (e.g., `001`, `002`) ensuring uniqueness *within* the same `A-[prefix]-[type]` combination. This allows multiple Workspace modals (01), Upsert modals (02), etc., for the same page prefix.
- **`[description]`:** A short, hyphenated description used for filename generation (e.g., `risk-workspace`, `contract-details`). The `modal_key` itself uses this description part.

**Example:**

- User enters Description: `Risk Workspace`
- Generated `modal_key`: `A-0300-01-001-risk-workspace`
- Generated `component_path`: `@pages/0300/components/modals/A-0300-01-001-RiskWorkspace.js` (PascalCase filename)

**Component File Location:** Component files should reside in the corresponding page's modal directory: `client/src/pages/[prefix]/components/modals/`.

**Uniqueness:** The combination of `prefix`, `type`, and `sequence` guarantees a unique identifier and filename, facilitating sorting and management within component folders.

## Best Practices

- **Keep Modals Focused:** Each modal should ideally perform a single, well-defined task.
- **Use `modalProps`:** Pass necessary data and callbacks to modals via the second argument of `openModal`. Avoid modals fetching their own configuration directly unless absolutely necessary.
- **Rebuild After Changes:** Remember to run `npm run build` after adding/deleting configurations or changing `modal_key`/`component_path` in the database.
- **Secure Service Key:** Ensure the `SUPABASE_SERVICE_KEY` in `.env` is kept secure and is available in the build environment.
- **RLS Policies:** Implement robust Row Level Security policies on the `modal_configurations` table to control read/write access appropriately for different user roles (`authenticated`, `admin`, etc.).
- **Manual Component Creation:** Adding a configuration to the database does *not* create the React component file. This must be done manually. Similarly, removing a configuration does not delete the file.

## Troubleshooting Guide

### Common Issues

1. **Modal Not Showing / "Modal with key ... not found":**
    - **Check Database:** Verify a configuration with the exact `modal_key` exists in the `modal_configurations` table.
    - **Check Build:** Ensure the application was rebuilt (`npm run build`) after the configuration was added/modified.
    - **Check Generated Registry:** Inspect `client/src/generated/modalRegistry.js` to confirm the key and import path are present and correct after the build.
    - **Check `is_legacy`:** Ensure the `is_legacy` flag is `false` in the database for React modals. The build script ignores legacy modals.
    - **Check Component Path:** Verify the `component_path` in the database points to the correct, existing React component file. Check for typos.
    - **Check Trigger:** Ensure the `openModal('your_modal_key')` call uses the exact, case-sensitive key.
    - **Check `display_name`:** Ensure the `display_name` column in `modal_configurations` is not null, as it's a required field.

2. **"No active user session found" Error in Components Relying on User Data (e.g., `0170-DuplicateModal.js` for page list):**
    - **Cause:** The component attempts to fetch user-specific data (like UI settings via `loadSupabaseSettings`) before a user session is fully established or restored.
    - **Production Fix:** Ensure authentication flow completes and session is available before rendering components that depend on it. Use loading states or conditional rendering.
    - **Development Workaround (Temporary):** If testing components in isolation without a full login flow, you might temporarily modify the component to bypass the user check and use hardcoded default values (e.g., a default `userId`, `userType`, `contractorName`) when calling services like `loadSupabaseSettings`. **Remember to remove this workaround before committing/deploying.** See `client/src/pages/0170-modal-management/components/0170-DuplicateModal.js` for an example implementation of this workaround in the `fetchPages` effect.

3. **Modal Management Page Shows No Modals:**
    - **Check Authentication:** Verify the user is logged into the application. Use browser dev tools to check Supabase session status.
    - **Check RLS Policy (SELECT):** Ensure an RLS policy exists on `modal_configurations` allowing the `authenticated` role (or the user's specific role) SELECT access. Example: `CREATE POLICY "Allow authenticated read access" ON public.modal_configurations FOR SELECT USING (auth.role() = 'authenticated');`
    - **Check Table Grants (SELECT):** Ensure the `authenticated` role has SELECT permission on the table: `GRANT SELECT ON TABLE public.modal_configurations TO authenticated;`
    - **Check Service (`fetchModalConfigurations`):** Add console logs within `modalConfigService.js` to see if the fetch is attempted and if it returns data or an error.

3. **Cannot Add/Delete Modals on Management Page:**
    - **Check Authentication/Role:** Verify the user is logged in and has the necessary role (e.g., 'admin').
    - **Check RLS Policy (INSERT/DELETE):** Ensure RLS policies exist allowing the user's role INSERT or DELETE access. Example: `CREATE POLICY "Allow admin insert access" ON public.modal_configurations FOR INSERT WITH CHECK (public.check_user_role(auth.uid(), 'admin_role_name'));` (Requires a role check function).
    - **Check Table Grants (INSERT/DELETE):** Ensure the user's role has INSERT/DELETE permission: `GRANT INSERT, DELETE ON TABLE public.modal_configurations TO admin_role_name;`
    - **Check Service (`addModalConfiguration`/`deleteModalConfiguration`):** Add console logs in the service functions to trace the request and response.

4. **Build Fails at `generateModalRegistry.js`:**
    - **Check `.env` File:** Ensure the `.env` file exists in the project root (`/Users/_PropAI/simandou-ai-bundle`) and contains the correct `SUPABASE_URL` and `SUPABASE_SERVICE_KEY`.
    - **Check Service Key:** Verify the Service Key is correct and has not been revoked in Supabase settings.
    - **Check Network/Supabase Status:** Ensure the build environment can reach the Supabase URL.
    - **Check Script Logs:** Look for specific error messages output by the `generateModalRegistry.js` script in the build logs.

### Debug Strategies

1. **Console Logging:** Add logs in `ModalProvider`, `useModal`, `modalConfigService.js`, `generateModalRegistry.js`, and specific modal components.
2. **Inspect Generated Registry:** Check the contents of `client/src/generated/modalRegistry.js` after a build.
3. **Network Monitoring:** Use browser dev tools to inspect requests made by `modalConfigService.js` to Supabase. Check status codes and responses.
4. **Supabase RLS Debugging:** Temporarily disable RLS or use the Supabase SQL editor to test policies with specific user roles/UIDs.
5. **Build Logs:** Carefully examine the output of `npm run build` for errors, especially from the `prebuild` step running `generateModalRegistry.js`.
