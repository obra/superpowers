# Simulation to Training Data Transformation

## Overview

The `scripts/transform_simulation_to_training.py` script transforms local Markdown files generated by the correspondence workflow simulator into training data records in the Supabase database. This bridges the gap between simulation outputs and the continual learning pipeline.

## Purpose

The correspondence workflow simulator generates specialist analyses and saves them as Markdown files in `agents/simulation/docs/correspondence-workflow/`. This script:

1. **Scans** the simulation output directory for Markdown files
2. **Parses** the files to extract specialist analyses
3. **Transforms** the data into training record format
4. **Saves** the records to the Supabase database

This enables the training pipeline to access real simulation data for model fine-tuning.

## Usage

### Basic Usage

```bash
python scripts/transform_simulation_to_training.py \
  --supabase-url https://your-project.supabase.co \
  --supabase-key your-service-role-key
```

### With Custom Simulation Directory

```bash
python scripts/transform_simulation_to_training.py \
  --supabase-url https://your-project.supabase.co \
  --supabase-key your-service-role-key \
  --simulation-dir path/to/simulation/outputs
```

### Dry Run Mode (Parse Only)

```bash
python scripts/transform_simulation_to_training.py \
  --supabase-url https://your-project.supabase.co \
  --supabase-key your-service-role-key \
  --dry-run
```

This parses files and displays the data without saving to the database.

## Command Line Arguments

| Argument | Required | Default | Description |
|----------|----------|---------|-------------|
| `--supabase-url` | Yes | - | Supabase project URL |
| `--supabase-key` | Yes | - | Supabase service role key |
| `--simulation-dir` | No | `agents/simulation/docs/correspondence-workflow` | Path to simulation output directory |
| `--dry-run` | No | `False` | Parse files without saving to database |

## Environment Variables

You can also use environment variables:

```bash
export SUPABASE_URL=https://your-project.supabase.co
export SUPABASE_SERVICE_ROLE_KEY=your-service-role-key

python scripts/transform_simulation_to_training.py
```

## How It Works

### 1. File Scanning

The script recursively scans the simulation directory for Markdown files:

```
agents/simulation/docs/correspondence-workflow/
├── 01_document_analysis/
│   ├── input_*.md
│   └── output_*.md
├── 02_information_extraction/
│   └── ...
├── 03_document_retrieval/
│   └── ...
├── 04_domain_specialist/  # Primary target
│   ├── output_civil_engineering_*.md
│   ├── output_structural_engineering_*.md
│   └── ...
├── 05_contract_management/
│   └── ...
└── 06_human_review/
    └── ...
```

### 2. File Parsing

For each Markdown file, the script:

1. Extracts the execution ID from the filename
2. Determines the workflow step type from the directory path
3. Parses content based on step type

**Domain Specialist Parsing** (primary focus):
- Extracts specialist type (e.g., `civil_engineering`)
- Extracts specialist analysis text
- Extracts confidence score
- Extracts citations
- Extracts correspondence text (if available)
- Calculates quality score based on analysis depth

### 3. Data Transformation

The parsed data is transformed into the database record format:

```python
{
    'simulation_id': 'exec_1234567890_abc123',
    'step_id': 'domain_specialist',
    'specialist': 'civil_engineering',
    'correspondence_text': 'Excerpt from correspondence...',
    'specialist_analysis': 'Technical analysis content...',
    'confidence': 0.85,
    'citations': '["citation1", "citation2"]',
    'quality_score': 85,
    'domain_relevance_score': 0.85,
    'model_version': 'simulation-1.0.0',
    'training_status': 'pending',
    'source_file': '/path/to/output_file.md'
}
```

### 4. Database Insertion

Records are saved to the `correspondence_training_data` table in Supabase:

- **Batch processing**: Records are inserted in batches of 50
- **Error handling**: Failed batches are logged but don't stop processing
- **Status tracking**: All records are saved with `training_status='pending'`

## Quality Score Calculation

The script calculates quality scores based on:

1. **Analysis Length**:
   - < 200 characters: 0.80 (80%)
   - 200-500 characters: 0.85 (85%)
   - > 500 characters with citations: 0.90 (90%)

2. **Specialist Type**:
   - Domain specialists: 0.80-0.90
   - Other steps: 0.65 (lower quality for non-specialist data)

3. **Citations**:
   - Presence of citations increases quality score

## Integration with GitHub Actions

The transformation script is automatically run in the GitHub Actions workflow:

```yaml
- name: Transform simulation data to training data
  id: transform-data
  run: |
    python scripts/transform_simulation_to_training.py \
      --supabase-url ${{ secrets.SUPABASE_URL }} \
      --supabase-key ${{ secrets.SUPABASE_SERVICE_ROLE_KEY }} \
      --simulation-dir "agents/simulation/docs/correspondence-workflow"
```

This ensures that simulation data is transformed before training begins.

## Workflow Integration

The transformation script fits into the continual learning pipeline:

```
1. Run Simulator
   └─> Generates Markdown files

2. Transform Data (this script)
   └─> Parses Markdown files
   └─> Saves to Supabase

3. Export Data
   └─> Queries Supabase for training data
   └─> Creates training dataset files

4. Train Models
   └─> Fine-tunes specialist models

5. Validate Models
   └─> Checks model quality

6. Activate Models
   └─> Deploys improved models
```

## Troubleshooting

### No Records Found

**Problem**: Script reports "No training records found to save"

**Solutions**:
1. Verify simulation directory path is correct
2. Check that Markdown files exist in the directory
3. Ensure files are in the expected format (with specialist information)
4. Run with `--dry-run` to see what's being parsed

### Database Connection Errors

**Problem**: "Failed to save to Supabase"

**Solutions**:
1. Verify Supabase URL is correct
2. Check that service role key has insert permissions
3. Test database connection manually
4. Check Supabase logs for detailed error messages

### Parsing Errors

**Problem**: "Failed to parse file"

**Solutions**:
1. Check Markdown file format
2. Verify specialist type is in the file content
3. Ensure analysis content is present
4. Run with `--dry-run` to see parsing results

## Best Practices

1. **Run After Simulation**: Always run the transformation script after running the simulator
2. **Dry Run First**: Use `--dry-run` to verify parsing before saving
3. **Check Logs**: Review the output logs for any warnings or errors
4. **Regular Updates**: Run regularly to keep training data up-to-date
5. **Quality Check**: Review transformed data quality scores
