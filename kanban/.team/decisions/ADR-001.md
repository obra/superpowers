## Context

We needed a data store for a team kanban tool embedded in a Git repository.
Options considered:
1. SQLite (used by vibe-kanban)
2. MD/JSON files in the repo

## Decision

Use Markdown files with YAML frontmatter for tasks and people, JSON for sprints.
All data lives in `.team/` directory, committed to the repo.

## Consequences

**Good:**
- No binary files, fully readable/diffable in git
- Claude Code agents can read and write `.team/*.md` directly
- No migration system needed
- Works offline, zero infrastructure

**Bad:**
- No transactions (last-write-wins on concurrent edits)
- No complex queries (must read all files to filter)
- Doesn't scale past ~1000 tasks without performance issues

## Mitigation

For team sizes < 100 people and projects < 500 tasks, file-based storage is fast enough.
If scale becomes an issue, introduce a build step that compiles .team/ into a SQLite cache.
